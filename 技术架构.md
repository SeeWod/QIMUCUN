# 中小企业 "AI全栈项目开发" 【技术架构】与【开发与部署指南】

本文件的使用方式：放在docs下，告诉AI让的代码生成都根据这个技术架构走，例如“请参阅docs下的“技术架构文档，并按照需求文档生成项目代码”

## 1. 适用范围 与 设计思想

- 人力资源：只有一名全栈开发者，并且要同时负责“前后端”，还要做“devOps”的时候
- 服务器资源：只有单台服务器，并且有多个应用需要部署与维护

总体来说，该规范适合于在资源严重贫乏，但期望用极少的资源，做尽可能多的事情，还要同时确保效率的情况。

没错：廉价、物美、高效、还要可持续、可扩展

## 2. 内容

本文档所定义的**技术架构**，即指按以下论述 开展全流程开发与运维的整体方案。

时至2026.3.29日，目前项目开发的统一流程为：**AI开发** + **全自动化提交、测试、部署的方式（CI/CD）** + **维护（基本也是开发者维护）**。所有开发流程框架都是这样的，在实践过程中，会根据项目的不同，在细节方面做相应的适应。在我们的架构中，由以下环节要做调整：

- 部署方式（我们采用了混合管理模式）
- 开发与生产多环境调试规范（前后端、各个部分保存自己的配置文件，代码提交之后的阶段，所有配置用环境变量传入——由CD自动在服务器根据（CICD的var/secret）产生.env）
- 目录规范（根据前两项的不同，进行调整）

### （1 部署论述

部署架构设计：单服务器、多应用情景，应采用 “混合管理模式” 的架构进行部署，即

- 先单独部署一个应用组，它应该包含这个服务器上所有的应用都需要共同用到的东西（中央网关nginx、mongoDB、redis、minio等），我们将它命名为“基础设施服务——infra”
- 再部署各个应用服务——app
  - 各个应用只分别部署自己的后端容器
  - 前端文件交由中央nginx统一管理，并给出自己应用对应的nginx配置文件，nginx会自动讲配置文件纳入
  - 数据库、redis、OSS等都接入infra配置的
  - 只有基础设施会建立主机端口映射，其他应用容器都通过连接到 基础设施网络，nginx通过网络DNS的容器名字转发请求到相关应用

### （2 开发与生产配置约定

在 frontend、backend及同类文件夹下分别保存两份配置文件: `xx.dev.xx`、`xx.prod.xx`.

其中`xx.dev.xx`用于开发环节，相关的配置项直接使用具体的值。`xx.prod.xx`用于部署环节，其内部的值都是 环境变量的占位符。（我们将所有环境变量的管理都放在这里集中管理，项目内只使用对象引用的语法）。使用：在开发环节，运行前后端分别使用其dev文件启动。生产环节，我们通过CI/CD产生所有所需的环境变量存储在.env。使用docker compose启动时，自行加载。

> 环境变量的包含关系：
>
> 所有代码目录的配置文件汇总
> ├── 产生的.env文件
> │ ├── CICD\_CONFIG.md配置的

> java项目存放示例：`application.dev.yml`，`application.prod.yml`
>
> js项目存放示例：放在src/config目录下：`config.dev.json`，`config.prod.json`

### （3 CI/CD流程：

注意：

- CI/CD要采用github的runners，然后通过SSH启动
- 推送触发目前采用main标签，不要用版本标签，使得任何推送都可以触发
- APP_DEPLOY_PATH 文件夹不存在，就直接在服务器上创建

#### 1.基础设施的CI/CD流程设计：

infra只有CD流程，没有CI流程。目前通过main标签，不要用版本标签，使得推送就可以直接CD
具体流程：

- 同样的devOps人员会在github上根据CICD\_CONFIG.md文件配置好所需的secrets/var
- 然后根据secrets/var及所需自动生成.env文件供docker compose使用。
- 通过SSH连接到远程服务器,在指定文件夹下执行docker-compose up -d --build -f \<infra\_PATH>/deploy/

#### 2.这各个应用的CI/CD流程设计

CI: CI的工作目前是:将前端打包后做成artifact、后端打包成用应用名命名的镜像,并放到指定的镜像仓库
CD: （手动触发）

- 根据secrets/var及所需自动生成.env文件供docker compose使用。
- 连接到远程服务器
  - 将本应用的nginx的配置文件放到 INFRA\_DEPLOY\_PATH/nginx/conf.d/<应用名>.conf
  - 将前端artifact的放到 INFRA\_DEPLOY\_PATH/static/<应用名>/ 下面（注意：）
  - 创建DEPLOY\_PATH目录，将本应用相关文件（.env,compose文件等）放到这里
- 重启基础设施 以及 启动自己的应用

> 注意：
>
> 将应用dist直接解压，将其中的内容直接放到 INFRA\_DEPLOY\_PATH/static/<应用名>/下面
>
> 同时要在应用的配置文件中将 静态文件指向这里

### （5 CICD\_CONFIG.md（常见字段及格式、命名约定）

该文件意在指明运维人员（或devOps人员），在代码提交到仓库后(开发阶段完成，开始CI/CD流程)该在CI/CD平台创建哪些、哪种配置字段

- 每一个字段以键值对给出，并且其后注释指明字段用途、字段类型（var/secrets）
- 字段以用途归类、整理

### infra的CICD\_CONFIG.md

```
基础
APP_NAME=INFRA_DEL  #var 服务名
APP_NETWORK_NAME=infra  #var infra的网络名字 
APP_DEPLOY_PATH=/var/www/infra   #var 部署infra存放的服务器路径
注意：中央nginx的容器与主机端口都是80：80，443：443，按默认的就好了，不用配置

远程登陆相关
SSH_HOST=      #secret 远程服务器的IP地址
SSH_PORT=22      #secret 远程服务器的SSH端口
SSH_KEY=      #secret 远程服务器的SSH密钥
SSH_USER=      #secret 远程服务器的SSH用户名

注意：mongoDB、redis、minio的端口都用默认值，不用配置，如有需要，后续再配置
## mongoDB连接所需：
MONGO_USER=      #secret mongoDB初始化的用户名
MONGO_PASS=      #secret mongoDB初始化的密码
MONGO_AUTH_SOURCE=      #var mongoDB初始化的认证源

## minio连接所需：
MINIO_ADMIN_URL=http://minio.yinghuonet.com  #var minio管理端URL
MINIO_ADMIN_USER=      #secret minio管理端的admin用户名
MINIO_ADMIN_PASS=      #secret minio管理端的admin密码
MINIO_SERVER_URL=http://s3.yinghuonet.com  #var minio的服务器URL
注：admin登录端口为9001，api为9000

## redis
略
```
